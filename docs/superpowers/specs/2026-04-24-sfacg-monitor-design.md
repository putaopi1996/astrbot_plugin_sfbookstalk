# SF 轻小说更新监控 AstrBot 插件设计

## 目标

开发一个 AstrBot 插件，用来监控一本 SF 轻小说是否更新，并通过 OneBot/aiocqhttp 适配器，把更新提醒发送到配置好的 QQ 群和 QQ 私聊联系人。

第一版支持类似 `https://book.sfacg.com/Novel/747572/` 的 SF 小说主页。插件会按配置的时间间隔访问小说链接。如果发现最新章节变化，就进入最新章节页面，读取章节名、更新时间、字数和预览内容，然后发送提醒消息。

## 已确认约束

- QQ 发送目标是 OneBot/aiocqhttp 这一类适配器。
- 收件人直接在配置里填写 QQ 群号和 QQ 号。
- 插件不要求用户先在群聊或私聊里发送绑定命令。
- 网络请求使用异步方式，优先用 `httpx`，不用同步的 `requests`。
- 运行时状态不能保存在插件源码目录里。
- 抓取、解析或发送失败时只记录日志，不能让定时循环崩掉。

## 配置项

插件会提供 `_conf_schema.json`，方便在 AstrBot WebUI 里配置。

第一版配置项：

- `novel_url`：SF 小说主页链接。
- `check_interval_minutes`：每隔多少分钟检查一次。
- `group_ids`：需要通知的 QQ 群号列表。
- `private_user_ids`：需要通知的 QQ 号列表。
- `notify_on_first_run`：首次运行时，是否把当前最新章节也发送出去。
- `preview_max_chars`：预览内容最多发送多少字。
- `request_timeout_seconds`：访问 SF 页面时的超时时间。
- `enable_llm_comment`：是否启用大模型点评，默认开启。
- `comment_prompt`：点评提示词模板。模板里可以使用作者名、小说名、章节名、更新时间、字数和预览内容等变量。
- `comment_fallback_text`：大模型调用失败时使用的兜底点评。

## 架构

### `SfNovelClient`

负责访问和解析 SF 页面。

它会提供这些能力：

- 访问小说主页，解析小说信息、作者名和最新章节链接。
- 访问最新章节页面，解析章节名、更新时间、字数和预览内容。
- 把 SF 的相对链接整理成完整链接。

解析时会尽量准备几个备用规则，减少 SF 页面小改版造成的影响。如果关键字段实在解析不到，就抛出可控的解析错误，由插件捕获并写日志。

### `UpdateState`

负责记住上一次已经通知过的章节。

至少保存：

- 上次已通知的章节链接或章节 ID。
- 上次检查时间。

优先考虑使用 AstrBot 的 KV 存储。如果这个方式在测试或兼容性上不顺手，就退一步使用 AstrBot 插件数据目录下的 JSON 文件。不会把运行时状态写进插件源码目录。

### `CommentGenerator`

负责调用 AstrBot 当前配置的大模型生成点评。

它会把章节信息填入 `comment_prompt`，再调用 AstrBot 当前 provider。提示词需要让模型重点点评两件事：

- 字数：这一章偏短、正常还是比较足。
- 预览内容：根据能看到的预览，给出简短、有分寸的剧情或阅读体验点评。

默认提示词会要求模型只输出一段简短中文点评，不剧透、不编造预览里没有的信息。如果大模型调用失败、超时或返回空内容，就使用 `comment_fallback_text`，并在日志里记录失败原因。

### `MessageBuilder`

负责生成要发送的文字。

基础消息格式按你的要求：

`（作者名）在xxx（时间）更新了字数为xxx的最新章节（章节名）`

消息还会附带：

- 最新章节链接。
- 截断后的预览内容。
- 由大模型生成的字数和预览内容点评。

### `SfMonitorPlugin`

AstrBot 插件主类，也就是 `Star` 类。

它负责：

- 读取插件配置。
- 初始化时启动异步定时任务。
- 插件卸载或停用时取消定时任务。
- 每轮检查时抓取最新章节，和本地状态比较；如果是新章节，就发送通知并更新状态。
- 发现新章节后，先调用大模型生成点评，再组装通知消息。
- 用 OneBot 的 `send_group_msg` 给 QQ 群发消息。
- 用 OneBot 的 `send_private_msg` 给 QQ 私聊发消息。

如果当前 AstrBot 没有可用的 OneBot 平台，插件会写一条清楚的错误日志，然后跳过本轮发送，等下一轮继续检查。

## 更新判断

插件会用最新章节链接或章节 ID 和已保存的上次通知记录做比较。

首次运行时：

- 如果 `notify_on_first_run` 为 `false`，只记录当前最新章节，不发送提醒，避免刚装插件就发旧章节。
- 如果 `notify_on_first_run` 为 `true`，会把当前最新章节也发送出去，并记录状态。

发现新章节后，插件会尝试给所有配置的群和联系人发送消息。如果部分发送失败，会记录失败目标。第一版可以在发送尝试结束后记录该章节，避免因为某个目标反复失败导致消息重复刷屏。

## 异常处理

需要处理的异常包括：

- 访问 SF 页面超时或 HTTP 失败。
- SF 页面结构变化导致解析失败。
- 大模型 provider 不可用、超时或返回空内容。
- OneBot 平台不可用。
- OneBot 给某个群或用户发送失败。
- 配置不合法，比如小说链接为空、QQ 号不是数字。

每种错误都要写日志，并带上足够的上下文。定时循环必须捕获异常，保证某一轮失败不会影响后续检查。

## 测试

实现阶段按测试先行来做。

测试覆盖：

- 用小说主页 fixture 解析出最新章节信息。
- 用章节页 fixture 解析出章节名、更新时间、字数和预览内容。
- `notify_on_first_run` 开启和关闭时的首次运行行为。
- 使用配置提示词生成大模型点评请求。
- 大模型调用失败时使用兜底点评。
- 生成符合配置的提醒消息。
- 通过假的 OneBot 客户端，验证会给配置的群和私聊发送消息。
- 不真实等待时间的情况下，测试“章节没变”和“发现新章节”的监控逻辑。

## 第一版不做的内容

- 同时监控多本小说。
- 支持 QQ 官方机器人主动发送。
- 通过聊天命令绑定收件人。
- 除 AstrBot 配置 schema 之外的额外 Web UI。
